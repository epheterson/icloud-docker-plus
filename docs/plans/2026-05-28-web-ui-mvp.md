# Web UI MVP — Implementation Plan

**Goal:** Give the running icloud-docker container a small built-in web UI so the user can re-authenticate Apple ID + 2FA from a phone (primary), confirm config + paths + sync health (secondary), and tail the recent log (tertiary). Ships as PR 9 to mandarons/icloud-docker and as icloud-docker-plus 0.6.0.

**Architecture:**
- Single Flask app embedded in the same Python process as the sync loop, running on a daemon thread spawned from `main.py`. Two threads, one process — they share the keyring + `/config/session_data` cookies + the log file. The web app **does not call into `sync.py`**; it shares state via the filesystem, like any external observer would.
- Phase 1 = read-only dashboard. Phase 2 = auth flow. Phase 3 = polish (config-gated, docker-entrypoint integration, README).
- LAN-trusted: no built-in login (you put it behind Authelia/Tailscale). Default port 8080 inside container; mandarons already `EXPOSE 80` for nothing, so we'd expose 8080 alongside it.

**Tech stack:** Flask 3.x (single new dep), stdlib threading, Jinja2 templates (built into Flask), inline CSS in a `base.html` (no separate static pipeline for v1). pytest + Flask test client for tests.

## Issues with the throwaway draft (already deleted) — addressed by this plan

These got caught in the research pass and inform the plan:

1. **Wrong `cookie_directory`.** Draft computed it via `os.path.dirname(get_username(...))` — nonsense. Correct value is `DEFAULT_COOKIE_DIRECTORY = "/config/session_data"` (verified in `src/__init__.py:24`). Plan task T6 uses the constant directly.
2. **2SA vs 2FA confusion.** `sync.py` gates on `requires_2sa`; icloudpy's 2FA path is a *strict subset* (hsaVersion==2). Plan task T8 handles the 2FA path; T8a documents the 2SA fallback as out-of-scope for v1 and surfaces an informative message instead.
3. **`trigger_2fa_push_notification` is in PR 1, not mandarons main.** mandarons/icloudpy 0.8.0 doesn't have it. The plan uses `hasattr` so the upstream PR works on either fork without depending on PR 1 merging first.
4. **Module-level `_PENDING_AUTH` dict.** Flask dev server is multi-threaded by default. A `threading.Lock` is enough for single-user, single-process. Plan task T8 locks the dict; T11 keeps single-process by running Werkzeug with `threaded=True, processes=1` (the default).
5. **Sync loop blocking during auth.** It does — sync.py logs "Retrying login at ..." and sleeps `retry_login_interval` (default 600s). Web UI auth flow writes to the same `/config/session_data` + keyring; **the next retry picks up the trusted session**. Plan documents this; no need to interrupt the sync loop.
6. **Log file path discovery.** Read from `config["app"]["logger"]["filename"]`; fall back to `/config/icloud.log`. Plan task T5.

```python
# src/web.py
import os
from flask import Flask, jsonify
from src import DEFAULT_CONFIG_FILE_PATH, ENV_CONFIG_FILE_PATH_KEY, read_config

def create_app(testing: bool = False) -> Flask:
    app = Flask(__name__)
    app.config["TESTING"] = testing

    @app.route("/api/health")
    def health():
        path = os.environ.get(ENV_CONFIG_FILE_PATH_KEY, DEFAULT_CONFIG_FILE_PATH)
        if not os.path.isfile(path):
            return jsonify({"state": "config_missing"}), 503
        return jsonify({"state": "ok"})

    return app
```

## Decisions (locked in 2026-05-28 09:11 PDT)

1. **Port:** 8080 inside container, compose maps `8080:8080`. mandarons' legacy `EXPOSE 80` stays untouched.
2. **Theme:** mockup 2–3 directions, render in a real browser via Chrome MCP, screenshot, pick the winner. Eric leaning Apple-ish, full discretion to me. New **Task T0** below covers this.
3. **Reverse proxy:** Cloudflare Tunnel terminates HTTPS + handles auth. Container speaks plain HTTP on `0.0.0.0:8080`. No CSRF token in v1 — Cloudflare access policy is the trust boundary.
4. **`web_ui.enabled` default:** **OFF.** Eric: "I don't wanna surprise people. I think this plus package, people are really gonna prefer over the existing ones." So the principle is *don't break existing vanilla setups by opening a new port*. README + UPSTREAM-PRS body advertise it as the headline feature; users opt in explicitly.

## Risks / known limitations of v1

- **2SA fallback unsupported** — if Apple drops you to the older device-list 2-step path (rare in 2026), the form will say "2SA fallback not supported in v1, please use the CLI." This affects ~0% of modern users.
- **Werkzeug dev server, not gunicorn** — fine for one user, not for high concurrency. Single-tenant by design.
- **No CSRF token on POSTs in v1** — relies on the user being behind Authelia/LAN. A `?token=` query param + a one-shot CSRF generated on dashboard load is a 30-line addition for v2.
- **Log tail is a re-read every refresh** — fine for 200 lines; if you start displaying 10k lines we'd need pagination or SSE streaming.
- **No "force re-sync now" button in v1.** That'd be nice but it needs a way for the web thread to signal the sync loop. Deferred to v2 — file-based "next run is forced" flag is easy enough.
